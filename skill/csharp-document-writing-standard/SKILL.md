# C# 项目文档编写规范
## 核心原则
文档核心是「记录为什么这么做，而不是怎么做的」，聚焦业务逻辑、设计决策、使用方式，不记录代码本身已明确的内容，确保文档与代码同步更新、精准可维护。

## 代码内XML注释规范（每次任务必须100%覆盖）
### 必须记录的内容
1.  类/接口：核心职责、适用场景、核心依赖关系
2.  方法：功能描述、入参含义、出参含义、可能抛出的异常、使用注意事项
3.  核心属性/字段：字段含义、约束规则、默认值、取值范围

### 绝对禁止记录的内容
1.  代码本身已清晰表达的实现细节（如「本方法循环遍历用户列表」）
2.  临时调试信息、个人笔记、无意义的占位注释

## 项目外Markdown文档规范（按需求复杂度同步更新）
### 每次任务必须更新的核心内容
1.  变更日志（CHANGELOG.md）：记录版本号、变更日期、本次实现的功能、新增/修改的类/接口、修复的问题、破坏性变更，与任务总结内容完全对齐
2.  业务逻辑文档：新增的核心业务规则、流程、权限要求、边界条件
3.  接口文档：新增/修改的API接口信息，优先通过Swagger自动生成，补充完整XML注释即可
4.  设计决策文档：本次需求涉及的架构调整、技术选型变更、非通用设计决策，必须记录「决策依据、替代方案、选型原因」

### 按需补充的可选内容
1.  环境配置说明：新增的环境变量、配置项、依赖的第三方服务
2.  部署注意事项：本次变更需要的数据库脚本、配置更新、前置操作
3.  故障排查指南：核心功能的常见问题、排查步骤、解决方案

### 绝对禁止记录的内容
1.  代码本身已清晰表达的实现细节、重复冗余的内容
2.  具体的代码实现，避免代码变更后需要同步维护文档
3.  临时调试信息、测试数据、个人笔记
4.  与本次需求无关的内容，不得修改不相关的文档
5.  数据库连接字符串、API密钥等敏感信息
6.  模糊无意义的描述（如「实现了相关功能、优化了部分代码」）

## 分场景适配规则
1.  简单需求：仅需更新代码内XML注释、变更日志
2.  中等需求：更新XML注释、变更日志、业务逻辑文档、接口文档
3.  复杂需求：全量覆盖所有必填内容，补充架构决策、部署指南等相关文档
