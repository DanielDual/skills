# C# 测试代码颗粒度规范
## 核心原则
以「方法级单元测试为核心，业务场景级集成测试为兜底」为基础，聚焦核心业务逻辑覆盖，不做无效的过度测试，也不出现核心场景测试遗漏。

## 各分层测试颗粒度与覆盖要求
### 1. 工具类/帮助类（Helpers/Utils/Extensions）
- 测试颗粒度：细粒度方法级单元测试，覆盖每个方法的所有分支场景
- 核心目标：确保工具方法的输入输出完全符合预期，无边界条件遗漏
- 覆盖率要求：核心方法分支覆盖率100%

### 2. 业务服务层（Services/Domain 层）
- 测试颗粒度：中粒度方法级单元测试+业务场景测试，核心方法覆盖全部分支，非核心方法覆盖主流程
- 核心目标：确保核心业务逻辑、规则、分支、异常场景全部符合需求
- 覆盖率要求：核心业务代码分支覆盖率≥80%

### 3. 控制器/接口层（Controllers/API 层）
- 测试颗粒度：粗粒度集成测试，覆盖接口的正常/异常核心场景，不重复测试Service层已覆盖的业务细节
- 核心目标：确保接口契约、入参校验、权限控制、响应格式符合规范
- 覆盖率要求：主流程覆盖率100%，核心异常场景全覆盖

### 4. 数据访问层（Repository/EF Core 层）
- 测试颗粒度：中粒度集成测试，针对核心数据操作场景，不测试ORM框架本身的原生功能
- 核心目标：确保数据增删改查、查询条件、事务处理逻辑正确
- 覆盖率要求：核心数据操作场景全覆盖

## 分场景适配规则
1.  细粒度测试强制场景：核心业务逻辑方法、全项目复用的工具类、包含大量分支判断的方法
2.  中粒度测试适用场景：控制器接口方法、数据访问层方法、非核心辅助业务方法
3.  粗粒度测试适用场景：跨多服务的完整业务流程、对外API接口契约测试

## 绝对禁止项
1.  禁止过度测试：不得为简单Getter/Setter、纯透传方法、框架生成代码编写单元测试，不得重复测试下层已覆盖的业务逻辑
2.  禁止测试不足：核心业务逻辑不得仅覆盖正常流程，必须覆盖异常场景、边界条件
3.  禁止脱离需求编写测试用例，所有测试必须对应需求的验收标准
4.  禁止编写无明确断言的无效测试用例
