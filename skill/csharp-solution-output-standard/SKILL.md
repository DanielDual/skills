# C# 需求实现方案输出规范
## 核心原则
方案输出以「无歧义可评估、可落地不超纲、定框架不定代码」为核心，下限满足需求评估要求，上限不提前编写具体代码实现，避免返工。

## 基础必达项（所有需求必须100%覆盖，无例外）
1.  明确需求核心目标与可量化验收标准，清晰定义「完成即合格」的判断依据
2.  明确技术选型与依赖说明：用到的.NET框架特性、内置类库；如需新增NuGet包，必须写明包名、兼容的.NET版本、核心用途
3.  明确代码结构与模块划分：贴合项目现有分层架构，写明新增/修改的类、接口、文件的完整路径
4.  步骤化描述核心业务逻辑与执行链路，不编写具体代码实现，仅定核心流转规则
5.  明确边界与风险处理：入参校验规则、异常场景处理方案、权限控制要求
6.  明确测试覆盖方案：写明测试的核心场景、测试层级与覆盖目标

## 复杂需求进阶补充项（跨模块开发、架构调整、核心业务重构需求必须补充）
1.  数据模型变更：明确新增/修改的数据库表、字段、索引、约束，适配EF Core等ORM框架规范
2.  接口契约定义：RESTful接口的路由、HTTP方法、请求/响应模型、状态码规范
3.  权限与安全设计：接口访问权限、鉴权方式、限流/防刷策略
4.  兼容性与回滚方案：是否存在破坏性变更、现有代码兼容性、回滚操作步骤

## 分场景颗粒度适配规则
1.  简单需求（单工具类、单接口、小Bug修复）：仅需覆盖基础必达项，简洁明确核心逻辑
2.  中等需求（单个业务模块、跨2-3个类的迭代）：基础必达项+接口契约、数据模型变更等核心进阶项
3.  复杂需求（跨服务开发、架构调整、核心业务重构）：全量覆盖所有必达项与进阶项，补充架构流转说明

## 绝对禁止项
1.  禁止提前编写具体代码实现（完整方法体、逐行业务代码），仅定义结构与逻辑框架
2.  禁止输出超出需求范围的额外功能设计，严格锁定需求边界
3.  禁止使用模糊无标准的描述（如「做好异常处理」），必须明确具体执行规则
4.  禁止省略需求验收标准、风险处理等核心必达项
